# Phase 02 Plan 01: CSRF Fix, Mail Error Handling, and Gunicorn Dependency Summary

**One-liner:** Fixed CSRF bypass caused by capitalized HTML input names, wrapped mail.send() in try/except with abort(500), and added gunicorn==21.2.0 to requirements.

## What Was Changed

### src/templates/contact.html

The contact form had HTML input `name` attributes with capitalized values (`Name`, `Email`, `Message`) while WTForms field names are lowercase (`name`, `email`, `message`). Because the field names didn't match, `form.validate_on_submit()` always returned False — the CSRF token generated by `{{ form.hidden_tag() }}` was never validated against the submitted data.

Changes made:
- `name="Name"` → `name="name"`
- `name="Email"` → `name="email"`
- `name="Message"` → `name="message"`
- `action="contact"` → `action="/contact"` (relative path to absolute)
- Removed invalid `target="./pages/views"` attribute from `<form>` tag

The `{{ form.hidden_tag() }}` line was left unchanged — it continues to render the CSRF token.

### src/main.py

Two changes:

1. Added `abort` to the Flask import: `from flask import Flask, abort, render_template, request, send_from_directory`

2. Wrapped `mail.send(msg)` in a try/except block. Previously, any SMTP failure (wrong credentials, network error, Gmail outage) produced an unhandled exception and returned a full Python stack trace to the browser. Now:
   - `app.logger.exception()` logs the full traceback server-side
   - `abort(500)` returns a generic HTTP 500 response with no stack trace exposed to the client

### src/requirements.txt

Added `gunicorn==21.2.0`. The Dockerfile uses `gunicorn app:app` in its CMD, but gunicorn was not listed as a dependency. This meant `pip install -r src/requirements.txt` would not install gunicorn, breaking container builds unless gunicorn was manually installed.

## Decisions Made

- **abort(500) over custom error template:** Flask's built-in 500 handler provides a generic response. Custom error templates can be added later; what mattered here was stopping stack trace leakage.
- **app.logger.exception() over logging.exception():** Using the Flask app logger ensures the traceback appears in the same log stream as other Flask messages, including any configured handlers (Cloud Logging on GAE).
- **gunicorn==21.2.0:** Latest stable release compatible with Python 3.12 at time of writing.
